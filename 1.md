# CONTROL F MASTER SPEC

Current master spec for the Control F demo app after the live-search bridge work.

---

## What This App Is Now

Control F is a polished demo app that now combines:
- real natural-language search into approved pension-fund websites
- real PDF discovery from those sites
- real Anthropic extraction from selected PDFs
- a shared evidence-rich Results view

It is no longer accurate to describe the app as purely hardcoded or purely static.

The current product is a hybrid:
- real search / scrape / extract flow
- demo-backed dashboard analytics and some tracker surfaces

That hybrid is intentional for the demo.

---

## The Product Story

The strongest demo story is:

1. User types a real pension-fund intelligence query
2. Control F finds the right approved pension-fund source page
3. Control F finds the relevant PDFs on that source page
4. User extracts a small targeted set of PDFs
5. Results show structured metrics plus source evidence

The product is for analysts who need fast monitoring across public pension documents without manually checking dozens of websites.

---

## Core User

Sarah is still the primary persona:
- fund-of-funds analyst
- tracks commitments, performance, terminations, and manager changes
- cares about new information, exact evidence, and trust

Product rules:
1. Don't make me check websites.
2. Show me what changed.
3. Let me verify instantly.
4. Don't get it wrong.

## Current Navigation

The app has five sections:

1. Search
2. Results
3. Dashboard
4. Trackers
5. Upload

Current meaning of each:

- Search
  Hero entry for natural-language live search.
- Results
  Shared output surface for live-search, uploaded PDF extraction, and scraped-URL extraction.
- Dashboard
  Demo analytics surface plus the real live tracker experience.
- Trackers
  Secondary archive/manage surface.
- Upload
  Manual extraction entrypoint that already works well.

---

## Current Hero Flow

### Search -> Dashboard -> Results

What happens now:

1. User types a real question or clicks a preset.
2. Search creates a live tracker.
3. The tracker animates into Dashboard.
4. Dashboard live tracker:
   - finds source pages
   - scans source page for PDFs
   - lets the user select PDFs
   - extracts metrics
5. Results page opens on the real extracted output.

This flow is now the highest-priority product path.

---

## Current Upload Flow

Upload is still important and now correctly feeds Results.

Supported modes:
- Upload PDF
- Scrape URL

Current behavior:
- Upload PDF extracts real metrics
- Scrape URL finds PDFs and extracts real metrics
- both now publish into the shared Results state

This means Upload is no longer an isolated island.

## Search Presets

Current presets in Search:
- `PSERS private markets IRR, TVPI, DPI, and NAV`
- `Minnesota SBI quarterly private markets performance`
- `SAMCERA private equity performance review`
- `PSERS new fund commitments and board approvals`

These should stay highly intentional and tied to real approved source families.

Good preset qualities:
- concrete
- pension-fund-specific
- likely to map to a strong source page
- likely to map to a small, relevant PDF set

Bad preset qualities:
- vague
- open-web dependent
- likely to return giant generic document libraries

## Results Truth Rules

The app must not confuse:
- extraction success
- answer quality

These are different.

Current result quality buckets:
- weak match
- partial match
- strong enough to review

Examples:
- performance query returns only broad asset-allocation rows -> weak match
- performance query returns `IRR`, `TVPI`, `NAV` but misses `DPI` -> partial match
- direct metric coverage of the query -> strong enough to review

The copy must always match the query intent.
Performance searches should never be described as failed commitment searches.

## Main Current Weaknesses

These are the next quality gaps, not broken plumbing:

1. Auto-retry on weak or partial match
2. Better preflight PDF quality judgment before Claude extraction
3. Multi-document handling for plural searches
4. Better ranking when a source page has many similar performance PDFs
5. More reliable extraction-cost guardrails before the user clicks extract

## Success Standard

A strong run looks like this:
- user asks a clear pension-fund question
- app finds a source page that makes sense
- app recommends a small, relevant PDF set
- extraction returns metrics that actually match the query
- Results default to the relevant metric family
- evidence is clear enough to trust quickly

That is the bar.
